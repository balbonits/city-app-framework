# Codebase Shape

A reference summarizing how five FAANG/MAANG companies organize their codebases, based on research conducted April 2026. Sources are primary engineering blogs, papers, talks, and conference materials where available, supplemented by named ex-employee accounts (Hacker News, personal blogs, journalism) where official documentation is sparse — flagged inline.

This document describes what is publicly evidenced. It does not prescribe a structure for any specific project.

---

## Two orthogonal axes

The conventional "monolith vs. microservices" framing collapses two independent questions:

1. **Repo structure:** monorepo (one tree) or polyrepo (many trees)
2. **Deploy unit:** single deployable artifact or many independent services

The five companies occupy three of the four cells in the resulting 2&times;2:

|              | **Single deploy** | **Many deploys** |
|--------------|---|---|
| **Monorepo** | Meta WWW (Hack/HHVM monolith on Sapling) [[1]](#meta) | Google (Piper + Bazel + Borg) [[2]](#google) |
| **Polyrepo** | Not represented in this research | Amazon (Brazil + version sets) [[3]](#amazon); Netflix (per-service repos + Spinnaker) [[4]](#netflix); Apple (~"thousands of individual projects" per ex-employee account) [[5]](#apple) |

Cell occupancy is evidenced as follows:

- **Meta WWW:** "Most of the company's code" lives in the `fbsource` Sapling monorepo; the WWW product surface is a Hack/HHVM monolith ([Sapling announcement, engineering.fb.com 2022](https://engineering.fb.com/2022/11/15/open-source/sapling-source-control-scalable/); [Hack announcement, engineering.fb.com 2014](https://engineering.fb.com/2014/03/20/developer-tools/hack-a-new-programming-language-for-hhvm/)).
- **Google:** Piper monorepo holds the bulk of Google's source — ~1B lines of code, ~86 TB, ~25,000 engineers committing ~16,000 changes/workday as of 2015 ([Potvin & Levenberg, CACM 2016](https://cacm.acm.org/research/why-google-stores-billions-of-lines-of-code-in-a-single-repository/)); engineers run hundreds-to-thousands of services orchestrated by Borg ([Verma et al., Borg paper](https://research.google/pubs/large-scale-cluster-management-at-google-with-borg/)).
- **Amazon:** Brazil organizes code into versioned packages composed via "version sets" — a manifest pinning every package version that goes into a build ([DoltHub, 2024](https://www.dolthub.com/blog/2024-11-06-brazil-on-dolt/); ex-Amazon engineer's gist, [terabyte/15a2d3d4...](https://gist.github.com/terabyte/15a2d3d407285b8b5a0a7964dd6283b0)). Each package is independently buildable and deployable.
- **Netflix:** "We commit code to a central Git repository" with each team picking the workflow that fits, distributed as ~234+ public OSS repos mirroring internal fragmentation ([Netflix Tech Blog, "How We Build Code"](https://netflixtechblog.com/how-we-build-code-at-netflix-c5d9bd727f15); service count widely cited as 700+ across multiple secondary summaries of Netflix talks).
- **Apple:** A current 13-year-tenured Apple engineer describes "thousands of individual projects" where "coordinated changes to multiple frameworks ... god damned impossible to do, and result in broken builds so often" ([HN comment 43254364, 2025](https://news.ycombinator.com/item?id=43254364), single ex-employee account). Apple's open-sourced Swift compiler stack is on a monorepo ([apple/llvm-monorepo-root](https://github.com/apple/llvm-monorepo-root)), but this covers compiler tooling, not OS source.

## What's documented at each company

### <a id="amazon"></a>Amazon

- **Repo / build:** Brazil + version sets (above). Interface versions (e.g. `1.1`, human-declared) are decoupled from build versions (e.g. `1.1.3847523`, machine-assigned); devs depend on the interface, version set resolves the build ([DoltHub](https://www.dolthub.com/blog/2024-11-06-brazil-on-dolt/)).
- **Service architecture:** Driven by the 2002 Bezos "API mandate," paraphrased by Steve Yegge from inside Amazon: all teams expose data and functionality through service interfaces; "no direct linking, no direct reads of another team's data store, no shared-memory model, no back-doors whatsoever"; all interfaces "must be designed from the ground up to be externalizable" ([Yegge, 2011 gist](https://gist.github.com/chitchcock/1281611)).
- **Deploy:** Apollo orchestrates fleet-wide rolling updates with auto-halting; ~50M deployments/year, &gt;1/sec average ([All Things Distributed, 2014](https://www.allthingsdistributed.com/2014/11/apollo-amazon-deployment-engine.html)). Pipelines glues Brazil → Apollo into per-app continuous delivery ([AWS Builders' Library — Continuous Delivery](https://aws.amazon.com/builders-library/going-faster-with-continuous-delivery/)).
- **Trunk-based development:** Confirmed by Clare Liguori, AWS principal SWE: "We do trunk-based development, so the pipeline only deals with the main branch. We rarely create branches." ([InfoQ, 2020](https://www.infoq.com/news/2020/07/continuous-delivery-amazon/))
- **Code review:** Per Liguori: "We prefer breaking up feature work into multiple small code reviews that can be merged and shipped to production individually." Code review is "the final manual checkpoint before automated deployment." ([InfoQ, 2020](https://www.infoq.com/news/2020/07/continuous-delivery-amazon/))
- **Feature flags:** Internal Weblab (private; not publicly documented at depth); AWS AppConfig is the productized public version ([AWS blog](https://aws.amazon.com/blogs/mt/using-aws-appconfig-feature-flags/)).
- **Two-pizza teams + ownership:** "No team should be big enough that it would take more than two pizzas to feed them" — typically &lt;10 people, single-threaded ownership end-to-end ([AWS Executive Insights](https://aws.amazon.com/executive-insights/content/amazon-two-pizza-team/)).
- **"You build it, you run it":** Vogels, ACM Queue 2006: "You build it, you run it." Dev team carries the pager, owns deploys ([ACM Queue](https://queue.acm.org/detail.cfm?id=1142065)).
- **Acknowledged costs (Yegge):** Pager escalations through ~20 service calls before finding the owner; "every single one of your peer teams suddenly becomes a potential DOS attacker" — requires quotas and throttling everywhere; observability burden — "monitoring and QA are the same thing" ([Yegge gist](https://gist.github.com/chitchcock/1281611)).

### <a id="meta"></a>Meta

- **Repo:** `fbsource` is a single Sapling monorepo — "tens of millions of files, tens of millions of commits, and tens of millions of branches" — with EdenFS virtual filesystem so working copies are fetched on demand. Sapling started in 2012 as a Mercurial extension; client open-sourced November 2022 ([engineering.fb.com](https://engineering.fb.com/2022/11/15/open-source/sapling-source-control-scalable/)). Linear, branch-free commit history; directory branching added October 2025 ([engineering.fb.com 2025](https://engineering.fb.com/2025/10/16/developer-tools/branching-in-a-sapling-monorepo/)).
- **Build:** Buck2 (open-sourced April 2023, Rust rewrite of Buck1) — "Thousands of developers at Meta are already using Buck2 and performing millions of builds per day," ~2x Buck1 speed ([engineering.fb.com](https://engineering.fb.com/2023/04/06/open-source/buck2-open-source-large-scale-build-system/)).
- **Deploy:** Quasi-continuous push from master since April 2017 — tens-to-hundreds of diffs batched and pushed every few hours ([engineering.fb.com, "Rapid release at massive scale"](https://engineering.fb.com/2017/08/31/web/rapid-release-at-massive-scale/)).
- **Code review:** Stacked diffs (not branch-based PRs), each diff a small self-contained change with explicit dependencies on diffs below ([Pragmatic Engineer](https://newsletter.pragmaticengineer.com/p/stacked-diffs-and-tooling-at-meta); [Jackson Gabbard](https://jg.gg/2018/09/29/stacked-diffs-versus-pull-requests/)). Internal stack: Phabricator → Sandcastle (CI) → Landcastle (lands diffs) → Butterflybot (review automation).
- **Feature flags:** Gatekeeper feature flags decouple code release from feature exposure ([engineering.fb.com 2017](https://engineering.fb.com/2017/08/31/web/rapid-release-at-massive-scale/)).
- **Service architecture:** Hack/HHVM monolith on the product surface ([engineering.fb.com 2014](https://engineering.fb.com/2014/03/20/developer-tools/hack-a-new-programming-language-for-hhvm/)); behind it, Twine orchestrates containers across ~1M machines per region ([engineering.fb.com](https://engineering.fb.com/2019/06/06/data-center-engineering/twine/); [OSDI '20 paper](https://www.usenix.org/system/files/osdi20-tang.pdf)).
- **"Move fast" arc:** Zuckerberg retired "move fast and break things" at F8 2014 in favor of "move fast with stable infra" — 2-year API stability guarantee, formal API versioning ([Wikimedia Commons photo of F8 slide](https://commons.wikimedia.org/wiki/File:Mark_Zuckerberg_-_Move_Fast_with_Stable_Infra.jpg)).
- **Operational ownership:** Production Engineers "share an on-call rotation with their engineering team and serve as an escalation contact for service incidents" ([Meta Careers](https://www.metacareers.com/jobs/877502507478895)). Whether all software engineers (not just PEs) carry pagers is less cleanly documented.
- **Acknowledged costs:** Naive git would take 45 minutes per command at fbsource scale (the reason Sapling exists); "any developer has access to most of the code" complicates ACLs; full-repo branching degrades `sl log`/`sl blame` for every user ([engineering.fb.com 2025](https://engineering.fb.com/2025/10/16/developer-tools/branching-in-a-sapling-monorepo/)).

### <a id="google"></a>Google

- **Repo:** Piper. ~1B LOC, ~86 TB, ~25,000 engineers committing ~16,000 changes/workday as of 2015 ([CACM 2016](https://cacm.acm.org/research/why-google-stores-billions-of-lines-of-code-in-a-single-repository/)); ~50,000 engineers, 80+ TB, 60–70k commits/workday per *Software Engineering at Google* ([SWE@Google ch.16](https://abseil.io/resources/swe-book/html/ch16.html)). Engineers don't check the repo out — most use **CitC** (Clients in the Cloud), a FUSE filesystem overlaying pending edits on the live repo head; &gt;80% of Piper users are on CitC ([CACM 2016](https://cacm.acm.org/research/why-google-stores-billions-of-lines-of-code-in-a-single-repository/)).
- **Build:** Blaze (internal) → Bazel (open-source). Hermetic execution, sandboxed actions, aggressive caching of all prior work ([bazel.build/about](https://bazel.build/about/intro)).
- **Trunk-based development:** "Everything ships to head; long-lived branches are rejected as 'pure overhead'" ([SWE@Google ch.16](https://abseil.io/resources/swe-book/html/ch16.html)). Incomplete features committed to trunk but disabled at runtime via flags or limited by build-system visibility rules.
- **Code review:** Three orthogonal sign-offs required for every change: peer LGTM, OWNER approval, Readability stamp from someone certified in that language ([SWE@Google ch.9](https://abseil.io/resources/swe-book/html/ch09.html), [ch.19](https://abseil.io/resources/swe-book/html/ch19.html)).
- **Test automation:** TAP processes &gt;50,000 unique changes and &gt;4 billion test executions per day ([SWE@Google ch.23](https://abseil.io/resources/swe-book/html/ch23.html)).
- **Deploy:** Borg orchestrates jobs across clusters of tens of thousands of machines ([Borg paper](https://research.google/pubs/large-scale-cluster-management-at-google-with-borg/)); Rapid + Sisyphus push from green-postsubmit out — hourly auto-pushes for pre-prod, exponential cluster expansion for user-facing services ([SRE Book — Release Engineering](https://sre.google/sre-book/release-engineering/)).
- **Operational ownership:** SRE Book documents shared on-call: "the developer teams of SRE-supported systems usually participate in a 24/7 on-call rotation" and "It is expected that Dev always carries at least some of the operational responsibilities" ([sre.google/workbook/on-call](https://sre.google/workbook/on-call/)).
- **Acknowledged costs:** *Software Engineering at Google* explicitly names: tooling tax (Piper, CitC, Blaze, TAP, Critique are all in-house investments most orgs cannot replicate); migration burden at scale (Python 2→3 took years); the **scaling paradox** — "as a codebase and the number of engineers grows, the largest atomic change possible counterintuitively *decreases*"; Hyrum's Law — every observable behavior of every API becomes load-bearing ([SWE@Google ch.1](https://abseil.io/resources/swe-book/html/ch01.html), [ch.22](https://abseil.io/resources/swe-book/html/ch22.html)).

### <a id="netflix"></a>Netflix

- **Repo:** Polyrepo — one git repo per service/library ([Netflix Tech Blog](https://netflixtechblog.com/how-we-build-code-at-netflix-c5d9bd727f15)). Public OSS org alone hosts 234+ repos ([github.com/Netflix](https://github.com/Netflix)). Service count is widely cited as 700+, trending toward 1,000.
- **Build:** Heavy Gradle wrapped in Nebula — Netflix's open-source plugin collection that "eliminate[s] boilerplate build logic and provide[s] sane conventions" ([nebula-plugins.github.io](https://nebula-plugins.github.io/)). Cross-team consistency enforced via `nebula.dependency-lock`, `nebula-dependency-recommender`, and `gradle-resolution-rules`.
- **Microservices model:** Adrian Cockcroft frames the unit as "loose coupling with clear APIs between teams, enabling independent deployment" — team boundary is the architectural boundary ([InfoQ Microservices Retrospective](https://www.infoq.com/presentations/microservices-netflix-industry/)).
- **Operational ownership ("Full Cycle Developers"):** "If you ship code that day, you are on-call to fix it if it breaks that night" ([InfoQ, 2018](https://www.infoq.com/news/2018/06/netflix-full-cycle-developers/)).
- **Deploy:** Local Nebula build → git → Jenkins → Bakery (immutable AMI) → Spinnaker rolls it out ([InfoQ, 2016](https://www.infoq.com/news/2016/03/How-Netflix-Builds-Code/)). Cited end-to-end time for a small change: ~16 minutes from check-in to multi-region deployment. Default rollout: red/black (parallel new cluster, traffic shift, old cluster kept for instant rollback). Risky changes go through Kayenta automated canary analysis ([Netflix Tech Blog](https://netflixtechblog.com/automated-canary-analysis-at-netflix-with-kayenta-3260bc7acc69)).
- **Feature flags:** Archaius supports dynamic feature toggles ([Archaius wiki](https://github.com/Netflix/archaius/wiki/Overview)).
- **Trunk-based development:** Secondary source only — "Netflix also uses a 'trunk-based development' approach, where changes are merged into the main codebase frequently" ([Medium summary](https://medium.com/@seyhunak/learning-best-practices-from-netflix-tech-stack-ci-cd-pipeline-a25a58f46711)). No primary Netflix-authored statement located.
- **Code review:** Secondary source only — "All pull requests must pass automated tests before they can be merged into the main codebase ... pull requests must be approved by a certain number of developers" ([Medium summary](https://medium.com/@seyhunak/learning-best-practices-from-netflix-tech-stack-ci-cd-pipeline-a25a58f46711)). No Netflix-authored description of review threshold/approval count located.
- **Resilience:** Chaos Monkey's stated philosophy: "Exposing engineers to failures more frequently incentivizes them to build resilient services" ([Netflix/chaosmonkey README](https://github.com/Netflix/chaosmonkey)). Hystrix in maintenance mode since Nov 2018; Netflix recommends `resilience4j` ([Netflix/Hystrix README](https://github.com/Netflix/Hystrix)).
- **Acknowledged costs:** "Dependency version management is incredibly complex, and the onus for debugging issues in the version resolution is on the consuming application" ([InfoQ — Scaling Developer Experience](https://www.infoq.com/articles/developer-experience-netflix/)). Log4Shell took "multiple days of developer crunch time" for off-paved-road services. Cockcroft's retrospective: industry adopters of microservices missed supporting practices (timeout budgets, library APIs hiding wire protocols) and got retry storms ([InfoQ Retrospective](https://www.infoq.com/presentations/microservices-netflix-industry/)).

### <a id="apple"></a>Apple

Apple publishes very little about internal engineering. Most of what follows is from named ex-employee accounts on Hacker News, journalism, and Apple's own job postings — not Apple-authored documentation.

- **Repo structure:** A current 13-year Apple engineer describes "thousands of individual projects," with cross-framework coordination "god damned impossible" and frequent broken builds ([HN comment 43254364, 2025](https://news.ycombinator.com/item?id=43254364), single ex-employee). Apple's open-sourced Swift compiler stack uses a monorepo ([apple/llvm-monorepo-root](https://github.com/apple/llvm-monorepo-root)), but that covers compiler tooling, not OS source.
- **Build system:** **XBS** is referenced in archived Apple opensource Libc Makefiles as the internal build infrastructure replacement ([opensource.apple.com Makefile.xbs](https://opensource.apple.com/source/Libc/Libc-320/Makefile.xbs.auto.html)). Brian Bergstrand's "Brief History" notes XBS was "very large, very involved, and not documented outside Apple" ([synack.net/~bbraun](http://www.synack.net/~bbraun/writing/oshistory.html)). For Xcode/SwiftPM and the Swift project itself: **llbuild**, modeled after Shake, Buck, and Ninja ([swift-llbuild repo](https://github.com/swiftlang/swift-llbuild)).
- **Code review tooling:** No public source identifies a current Apple internal code-review tool by name. Workflow appears Radar-mediated per ex-employee account: "you would then get radars that mentioned the code-name and explained a bug that you had to fix for the feature, but you had to fix the bug blinded and send the bug back to verify" ([HN 21221589](https://news.ycombinator.com/item?id=21221589), single ex-employee under Bertrand Serlet). Public Swift project uses GitHub PRs with contributor ladder ([swift.org/contributing](https://www.swift.org/contributing/)).
- **Deploy / release:** Annual major-version trains (iOS X, macOS X) with point releases. Internal "Livability" app buckets features by intended release window — Spring/WWDC/Fall/Winter — with status tags Under Development / Code Complete / Preview Ready / Feature Complete ([AppleInsider, 2024](https://appleinsider.com/articles/24/06/26/how-apples-software-engineering-teams-manage-and-test-new-operating-system-features-ahead-of-launch)). Build numbering decoded by saagarjha (e.g., macOS 20A4300b = major 20, train A, audience 4 = developer seed) implies continuous build numbering per "train" with forks ([HN 23915031](https://news.ycombinator.com/item?id=23915031)).
- **Feature flags:** Livability tags features for staged release windows (above). Internal builds carry feature flags for features 2+ years out ([AppleInsider, 2024](https://appleinsider.com/articles/24/06/26/how-apples-software-engineering-teams-manage-and-test-new-operating-system-features-ahead-of-launch)).
- **Trunk-based development:** No public Apple-authored or named-insider source describes Apple's branching model for the iOS/macOS internal codebase.
- **Operational ownership:** Apple has dedicated SRE roles for iCloud — "Apple Service Engineering's iCloud SRE team is looking for Site Reliability Engineers to build and run the services that hundreds of millions of customers use every day" ([Apple Jobs](https://jobs.apple.com/en-us/details/200618403-0836/senior-site-reliability-engineer-apple-services-engineering-ase-icloud)). This suggests a Google-style split rather than Netflix-style full-cycle. No source confirms feature engineers carry production pagers.
- **Service architecture (iCloud):** Stateless service tier on FoundationDB (with Record Layer) and Cassandra at scale — 300,000+ Cassandra nodes, hundreds of petabytes ([engineerscodex summary of Apple's papers](https://read.engineerscodex.com/p/how-apple-built-icloud-to-store-billions)). Hosting mixed across own data centers, AWS, and GCP since 2016 ([Wikipedia: iCloud](https://en.wikipedia.org/wiki/ICloud)).
- **Org model:** Functional org (engineering, design, ops, marketing, retail under SVPs) — adopted by Jobs in 1997 and still the core model ([Panmore analysis](https://panmore.com/apple-inc-organizational-structure-features-pros-cons); HBR's "How Apple Is Organized for Innovation" summary).
- **Acknowledged costs:** Documented cross-team friction in the Siri/Apple Intelligence breakdown ([AppleInsider, 2025](https://appleinsider.com/articles/25/04/10/behind-the-scenes-siris-failed-ios-18-upgrade-was-a-decade-long-managerial-car-crash)) and Apple Maps' early failures partly attributed to privacy rules limiting cross-team data sharing ([AppleInsider, 2022](https://appleinsider.com/articles/22/04/15/apples-user-privacy-stance-has-caused-problems-for-internal-engineering-teams)). Engineer-described build breakage from cross-framework changes ([HN 43254364](https://news.ycombinator.com/item?id=43254364), single ex-employee).

## Costs documented per cell

### Monorepo + single deploy (Meta WWW)
Naive git would take 45 minutes per command at fbsource scale; "any developer has access to most of the code" complicates ACLs ([engineering.fb.com 2025](https://engineering.fb.com/2025/10/16/developer-tools/branching-in-a-sapling-monorepo/)). Mitigation requires permanent investment in dedicated infra teams.

### Monorepo + many deploys (Google)
Tooling tax — Piper, CitC, Blaze, TAP, Critique are all in-house investments most orgs cannot replicate ([SWE@Google ch.22](https://abseil.io/resources/swe-book/html/ch22.html)). The scaling paradox: at sufficient scale, "the largest atomic change possible counterintuitively *decreases*" ([SWE@Google ch.1](https://abseil.io/resources/swe-book/html/ch01.html)). Multi-year migrations (Python 2→3) even with LSC infrastructure. Hyrum's Law exposure: with so many internal consumers, every observable behavior becomes load-bearing.

### Polyrepo + many deploys (Amazon, Netflix, Apple)
- Amazon (Yegge): ~20-service-call pager bounces; "every single one of your peer teams suddenly becomes a potential DOS attacker"; observability burden ([Yegge gist](https://gist.github.com/chitchcock/1281611)).
- Netflix: "dependency version management is incredibly complex, and the onus for debugging issues in the version resolution is on the consuming application" ([InfoQ](https://www.infoq.com/articles/developer-experience-netflix/)). Log4Shell took "multiple days of developer crunch time" for off-paved-road services.
- Apple (ex-employee): cross-framework coordinated changes "result in broken builds so often" ([HN 43254364](https://news.ycombinator.com/item?id=43254364)).
- Common pattern: requires team-built dependency-management tooling (Brazil version sets, Nebula plugins) to remain livable.

### Polyrepo + single deploy
Not represented in the five companies researched.

## What this research does and doesn't establish

**Well-established:**
- Each of the five companies' cell occupancy, with primary sources for four (Amazon, Meta, Google, Netflix) and a single ex-employee account for the fifth (Apple).
- Per-company tooling investment, all five with at least one primary-source citation.
- Per-cell costs, all three occupied cells with documented examples.
- Continuous-or-near-continuous deploy at four of five companies (Apple is the exception).
- Feature flags at all five.

**Less established:**
- Apple's internal repo topology — single ex-employee HN account vs. silence from Apple itself.
- Apple's code review tool — not publicly named; ex-employee accounts describe Radar-mediated workflow.
- Apple's branching model — no primary or named-insider source for the iOS/macOS codebase.
- Apple's operational-ownership model (full-cycle vs. split) — dedicated SRE roles exist in job postings, but no source confirms whether feature engineers carry pagers.
- Netflix's trunk-based development and code review specifics — only secondary sources located.
- Meta's distinction between Production Engineers' oncall and software engineers' oncall — PE oncall well-documented, broader SWE oncall less so.
