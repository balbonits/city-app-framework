## Development Process

### Getting Started
1. Fork the repository
2. Create a feature branch: `git checkout -b feature/amazing-new-feature`
3. Read the relevant documentation in `/docs`
4. Make your changes following our coding standards
5. Test your changes thoroughly
6. Submit a pull request

### Branch Naming Convention
- `feature/description` - New features
- `fix/description` - Bug fixes  
- `docs/description` - Documentation updates
- `refactor/description` - Code refactoring
- `test/description` - Test improvements

### Commit Message Format
```
feat: add user authentication system

- Implement JWT-based authentication
- Add login/logout functionality
- Include password strength validation
- Add comprehensive tests

🤖 Generated with Claude Code (https://claude.ai/code)

Co-Authored-By: Claude <noreply@anthropic.com>
```

### Code Standards
- **TypeScript**: Use strict mode, avoid `any` types
- **Testing**: Include tests for all new functionality
- **Documentation**: Update relevant docs for changes
- **AI Context**: Add AI-friendly context to complex code
- **Accessibility**: Follow WCAG 2.1 AA guidelines
- **Performance**: Consider mobile-first performance

## Pull Request Process

### Before Submitting
- [ ] Code follows project style guidelines
- [ ] All tests pass locally
- [ ] Documentation has been updated
- [ ] AI context has been added where appropriate
- [ ] Changes have been tested on multiple platforms (if applicable)

### PR Description Template
```markdown
## What this PR does
Brief description of the changes

## Type of change
- [ ] Bug fix (non-breaking change which fixes an issue)
- [ ] New feature (non-breaking change which adds functionality)
- [ ] Breaking change (fix or feature that would cause existing functionality to not work as expected)
- [ ] Documentation update

## Testing
- [ ] Unit tests added/updated
- [ ] Integration tests added/updated
- [ ] Manual testing completed
- [ ] AI integration tested (if applicable)

## Checklist
- [ ] My code follows the project's style guidelines
- [ ] I have performed a self-review of my own code
- [ ] I have commented my code, particularly in hard-to-understand areas
- [ ] I have made corresponding changes to the documentation
- [ ] My changes generate no new warnings
- [ ] I have added tests that prove my fix is effective or that my feature works
- [ ] New and existing unit tests pass locally with my changes
```

## Code Review Process

### What Reviewers Look For
- Code quality and maintainability

- Documentation completeness

### Review Timeline
- Initial review: 2-7 days
- Follow-up reviews: 1-3 days

## Getting Help

### Questions?
- Check existing [GitHub Issues](https://github.com/[username]/city-app-framework/issues)
- Review the [documentation](/docs)
- Ask questions in new GitHub Issues with the `question` label

### Bug Reports
Use this template:
```markdown
**Bug Description**
A clear and concise description of what the bug is.

**To Reproduce**
Steps to reproduce the behavior:
1. Run command '...'
2. See error

**Expected Behavior**
A clear and concise description of what you expected to happen.

**Environment:**
- OS: [e.g. macOS, Windows, Linux]
- Node.js version: [e.g. 18.17.0]
- Framework version: [e.g. 1.0.0]

**Additional Context**
Add any other context about the problem here.
```

### Feature Requests
Use this template:
```markdown
**Feature Description**
A clear and concise description of what you want to happen.

**Use Case**
Describe the problem you're trying to solve and how this feature would help.

**Proposed Solution**
Describe the solution you'd like to see implemented.

**Additional Context**
Add any other context or screenshots about the feature request here.
```
